# מטבח החסד - מסד נתונים (Supabase / PostgreSQL)

מיגרציות סכימת מסד הנתונים למערכת ניהול מטבח החסד, לפי מסמך האיפיון
`איפיון_מערכת_מטבח_החסד.md`.

> הסטאק שנבחר: **PostgreSQL (Supabase)** למסד, **Node + Express** לשרת,
> **React + Vite + Tailwind (RTL)** לפרונט.
>
> מוסכמה: שמות טבלאות ועמודות **באנגלית** ליציבות טכנית; כל התוויות שהמשתמש
> רואה הן **עברית מלאה (RTL)** בשכבת הממשק.

---

## מבנה התיקייה

```
supabase/
├── migrations/          # סכימת המסד - מריצים לפי הסדר
│   ├── 00000000000000_extensions_enums_helpers.sql
│   ├── 00000000000001_users_customers.sql
│   ├── 00000000000002_catalog.sql
│   ├── 00000000000003_shabbatot.sql
│   ├── 00000000000004_orders.sql
│   ├── 00000000000005_payments_refunds.sql
│   ├── 00000000000006_inventory_suppliers.sql
│   ├── 00000000000007_volunteers.sql
│   ├── 00000000000008_expenses_settings_audit.sql
│   └── 00000000000009_auto_enable_rls.sql
├── seed/
│   └── seed.sql         # נתוני ברירת מחדל (סעודות, קטגוריות, הגדרות, מיילים)
```

**חשוב:** המיגרציות תלויות זו בזו ו**חייבות לרוץ לפי סדר שם הקובץ** (00 → 08).
ה-`seed.sql` רץ **אחרי** כל המיגרציות.

## איך מריצים (כשתפתחי פרויקט Supabase)

### אפשרות א׳ - Supabase CLI (מומלץ)

```bash
# התקנה חד-פעמית
npm install -g supabase

# קישור לפרויקט (מזהה הפרויקט מ-Dashboard > Settings)
supabase link --project-ref <PROJECT_REF>

# הרצת כל המיגרציות
supabase db push

# הרצת ה-seed
psql "<CONNECTION_STRING>" -f supabase/seed/seed.sql
```

### אפשרות ב׳ - הדבקה ידנית ב-SQL Editor

1. ב-Supabase Dashboard → **SQL Editor**.
2. פתחי כל קובץ מיגרציה **לפי הסדר** (00 → 08), הדביקי והריצי.
3. לבסוף הריצי את `seed/seed.sql`.

### אפשרות ג׳ - psql ישירות

```bash
# מריץ את כל המיגרציות לפי סדר + seed
for f in supabase/migrations/*.sql; do psql "<CONNECTION_STRING>" -f "$f"; done
psql "<CONNECTION_STRING>" -f supabase/seed/seed.sql
```

## החלטות ארכיטקטורה מרכזיות

- **מחיקה רכה בכל מקום** (`is_active`) במקום `DELETE`, לפי סעיף 32. פריט
  לא-פעיל נעלם מבחירה חדשה אך נשמר בהיסטוריה.
- **סכומים ומחירים "קפואים" בהזמנה** (סעיף 15.3): `order_meal_slots`,
  `order_meals`, `order_extras` שומרים snapshot של מחיר/שם - שינוי מחירון עתידי
  לא משנה הזמנות קיימות.
- **מספור הזמנות שנתי רץ** דרך `allocate_order_number(year)` - פונקציה אטומית
  שמונעת כפילויות במקרה של הזמנות במקביל (סעיף 10.3, פורמט `YYYYNNNN`).
- **המלאי אינו מופחת אוטומטית** (סעיף 25.4): הפחתה נרשמת כתנועה ב-
  `inventory_movements` רק לאחר אישור ידני, עם תיעוד לפני/אחרי.
- **הערות פנימיות מסווגות לפי סוג** (`internal_note_type`) כדי שיופיעו בדוח
  הנכון בתיק השבת (מטבח/אריזה/שינוע/כספי) - סעיף 20.3.

## מה עדיין לא נכלל (כוונה - לשלב הבא)

- **RLS מופעל אך ללא policies** - מיגרציה `09` מפעילה RLS על כל הטבלאות
  (נעול כברירת מחדל). כרגע הגישה מיועדת דרך **שרת Node עם service_role key**
  שעוקף RLS. אם בעתיד הפרונט יגש ישירות למסד - יהיה צורך להוסיף **policies**
  פרטניות לפי תפקיד.
- **Supabase Auth** - כרגע `app_users.auth_uid` מוכן לקישור עתידי.
- **חישובי כמויות/אריזה** - לוגיקה עסקית, מקומה בשרת (services), לא במסד.
- **תצוגות (views) לדוחות** - יתווספו לפי הצורך (דוח מטבח, קניות, שינוע).
